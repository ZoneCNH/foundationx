# kernel 自动化依赖升级 Goal 执行方案 v1.0

> 适用仓库：`github.com/ZoneCNH/kernel`  
> 方案对象：L0 内核库 / Kernel Primitives / 标准库扩展  
> 默认上游标准：`https://github.com/ZoneCNH/xlib-standard`  
> 执行协议：Goal Runtime Prompt v3.1  
> 执行模式：Full Governance / Small Batch Execution  
> 日期：2026-06-02  
> 目标版本建议：`kernel v0.2.1` patch sync

---

## 0. 结论先行

`kernel` 必须落地自动化依赖升级，但不能把自动化理解成“自动追新版本”。  
`kernel` 是 L0 底座，升级风险会被所有 L1/L2 基础库和 `x.go` 放大，因此必须采用：

```text
发现升级 → 风险分级 → 创建 PR → Harness Gate → Evidence → 自动合并/人工审批 → 回滚/复盘
```

本方案推荐以 4 个 PR 落地：

| PR | 主题 | 目标 |
|---|---|---|
| PR-1 | Align Harness Gate | 修复 `lint/security` 缺工具跳过问题，改为 fail-fast |
| PR-2 | Add Dependency Bots | 接入 Renovate + Dependabot security fallback |
| PR-3 | Add Dependency / Standard Drift Gate | 增加 `dependency-check` 和 `standard-drift-check` |
| PR-4 | Extend Evidence Manifest | 把依赖清单、工具版本、升级证据写入 release manifest |

最终闭环：

```text
Renovate / Dependabot / standard-drift-check
  ↓
Dependency PR
  ↓
GOWORK=off make ci
  ↓
GOWORK=off make release-final-check
  ↓
release manifest + dependency evidence
  ↓
patch auto-merge / minor-major manual review
```

## 1. 问题的底层本质

表层问题是：

> 如何让 kernel 自动升级依赖？

底层问题是：

> 如何在不破坏 L0 契约、不污染下游依赖图、不降低安全与 Evidence 标准的前提下，让低风险升级自动流动，高风险升级自动阻塞。

`kernel` 不只是普通 Go module。它提供：

- `errx`
- `timex`
- `lifecycx`
- `retryx`
- `healthx`
- `obsx`
- `validx`
- `syncx`
- `versionx`
- `contracttest`

这些能力是所有上层库的共同底座。  
所以它的自动化升级必须比普通业务项目更严格。

---

## 2. 不可再拆解的基本真理

| 编号 | 基本真理 | 执行含义 |
|---|---|---|
| T-001 | L0 API 是长期契约 | 自动升级不能破坏公共 API、contract、schema |
| T-002 | 自动升级必须可回滚 | 每个 dependency PR 必须有 rollback plan |
| T-003 | 没有 Evidence 不允许合并 | PR 必须生成 dependency evidence 与 release manifest |
| T-004 | patch 可以自动化，major 不能自动化 | major 默认 `needs-review` |
| T-005 | L0 不能反向依赖 L1/L2/x.go | dependency-check 必须检查边界 |
| T-006 | 缺工具不能降级 | `golangci-lint` / `govulncheck` 不存在时必须失败 |
| T-007 | xlib-standard 是标准源 | kernel 必须能检测标准漂移 |
| T-008 | 依赖升级是治理事件 | 需要 Issue、Gate、Evidence、Review、Retro |

---

## 3. 被误认为真理的常见假设

| 假设 | 问题 | 正确裁决 |
|---|---|---|
| Dependabot 打开就算自动化 | 只解决发现，不解决契约验证 | 必须接 Harness + Evidence |
| patch 一定安全 | L0 的 patch 也可能破坏行为 | patch 可自动合并，但必须 Gate 全绿 |
| CI 通过就足够 | 没有 dependency diff / manifest 证据 | 必须写入 release evidence |
| 机器人可以自动合并所有依赖 | major、internal L0、standard sync 风险高 | 高风险强制人工审批 |
| kernel 只需要关注 go.mod | GitHub Actions、工具链、标准源也会变 | 必须覆盖 gomod/actions/toolchain/standard |

---

## 4. 可以被打破的限制

| 旧限制 | 新方案 |
|---|---|
| 人工检查依赖更新 | Renovate / Dependabot 自动发现 |
| 人工判断是否安全 | packageRules + labels + Gate 自动分级 |
| 手工记录升级结果 | release manifest 自动写入 dependency evidence |
| xlib-standard 变更后人工问 kernel 是否同步 | `standard-drift-check` 自动发现 |
| 失败后靠经验回滚 | 每个 PR 要有 rollback plan |
| Gate 可选执行 | `release-final-check` 作为最终门禁 |

## 6. Spec

### SPEC-kernel-dependency-upgrade-v1.0

#### REQ-001：Gate 严格性

`kernel` 的 required gate 不得在缺工具时降级为 skip。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-001-001 | `golangci-lint` 缺失时 `make lint` 失败 |
| AC-001-002 | `govulncheck` 缺失时 `make security` 失败 |
| AC-001-003 | CI 明确安装 required tools |
| AC-001-004 | `GOWORK=off make ci` 通过 |

#### REQ-002：自动依赖发现

仓库必须能自动发现 Go modules 与 GitHub Actions 依赖更新。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-002-001 | 根目录存在 `renovate.json` |
| AC-002-002 | 存在 `.github/dependabot.yml` |
| AC-002-003 | Renovate 启用 `gomod` 与 `github-actions` managers |
| AC-002-004 | major update 不自动合并 |
| AC-002-005 | `github.com/ZoneCNH/kernel` / `github.com/ZoneCNH/xlib-standard` 相关依赖不自动合并 |

#### REQ-003：依赖差异验证

每次依赖升级必须产生机器可验证的 dependency evidence。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-003-001 | 存在 `scripts/check_dependency_diff.sh` |
| AC-003-002 | `make dependency-check` 可执行 |
| AC-003-003 | 生成 `release/dependency/modules.txt` |
| AC-003-004 | 生成 `release/dependency/updates.txt` |
| AC-003-005 | `go mod tidy` 后 `go.mod/go.sum` 不应有未提交差异 |

#### REQ-004：标准漂移检测

`kernel` 必须能检测 `xlib-standard` 标准源变化是否需要同步。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-004-001 | 存在 `.standard-sync.yaml` |
| AC-004-002 | 存在 `scripts/check_standard_drift.sh` |
| AC-004-003 | `make standard-drift-check` 可执行 |
| AC-004-004 | 禁止残留 `pkg/templatex` / `package templatex` / `github.com/ZoneCNH/baselib-template` |
| AC-004-005 | 生成标准同步报告 |

#### REQ-005：Evidence Manifest 扩展

release manifest 必须记录依赖清单、可升级依赖、工具版本和 hash。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-005-001 | manifest 包含 `dependencies` |
| AC-005-002 | manifest 包含 `tools` |
| AC-005-003 | manifest 包含 dependency hash |
| AC-005-004 | `make release-evidence-check` 校验依赖证据 |
| AC-005-005 | `make release-final-check` 通过 |

## 7. Design

### 7.1 目标架构

```text
GitHub / Go Proxy / xlib-standard
  ↓
Renovate / Dependabot / standard-drift-check
  ↓
Dependency PR
  ↓
Makefile Gates
  ├─ fmt
  ├─ vet
  ├─ lint
  ├─ test
  ├─ race
  ├─ boundary
  ├─ security
  ├─ contracts
  ├─ dependency-check
  ├─ docs-check
  └─ release-final-check
  ↓
Evidence
  ├─ release/manifest/latest.json
  ├─ release/dependency/modules.txt
  ├─ release/dependency/updates.txt
  └─ standard sync report
  ↓
Decision
  ├─ patch auto-merge
  ├─ minor review
  ├─ major block
  └─ rollback
```

### 7.2 自动合并规则

| 类型 | 自动合并 | 原因 |
|---|---:|---|
| Go patch | 是 | 低风险，但必须 Gate 全绿 |
| GitHub Actions patch/minor | 是 | 可自动化，但需要 CI |
| Go minor | 否 | 可能有行为变化 |
| Go major | 否 | 破坏性风险 |
| security patch | 条件自动 | 低风险安全补丁可自动，高风险需审批 |
| kernel/internal module | 否 | L0 影响下游 |
| xlib-standard sync | 否 | 标准源变化必须 review |

### 7.3 Branch Protection

必须设置：

```text
main:
  require_pull_request: true
  require_status_checks: true
  required_checks:
    - kernel-ci / ci
  require_up_to_date: true
  restrict_force_pushes: true
  restrict_deletions: true
```

### 8.4 新增 scripts/check_dependency_diff.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"

mkdir -p release/dependency

echo "== go module list =="
GOWORK=off go list -m all | tee release/dependency/modules.txt

echo "== available module updates =="
GOWORK=off go list -m -u all | tee release/dependency/updates.txt

echo "== go mod tidy check =="
GOWORK=off go mod tidy

if ! git diff --exit-code -- go.mod go.sum; then
  echo "ERROR: go.mod/go.sum changed after go mod tidy"
  exit 1
fi

echo "dependency diff check passed"
```

### 8.6 新增 scripts/check_standard_drift.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"

[ -s ".standard-sync.yaml" ] || {
  echo "ERROR: .standard-sync.yaml missing"
  exit 1
}

mkdir -p release/standard-sync

{
  echo "# kernel standard sync report"
  echo
  echo "- generated_at: $(date -u '+%Y-%m-%dT%H:%M:%SZ')"
  echo "- source: ZoneCNH/xlib-standard"
  echo "- target: ZoneCNH/kernel"
  echo
  echo "## Local forbidden token check"
} > release/standard-sync/latest.md

for token in \
  "pkg/templatex" \
  "package templatex" \
  "github.com/ZoneCNH/baselib-template"; do
  if grep -R "$token" . \
    --exclude-dir=.git \
    --exclude-dir=release \
    --exclude=".standard-sync.yaml"; then
    echo "ERROR: forbidden standard/template token found: $token"
    exit 1
  fi
done

echo "- forbidden template tokens: passed" >> release/standard-sync/latest.md
echo "standard drift local check passed"
```

## 13. Rollback Protocol

任一自动依赖升级 PR 合并后出现失败：

```text
1. 标记 regression/dependency
2. revert merge commit
3. 创建 rollback issue
4. 记录失败 dependency、版本、失败 Gate、影响范围
5. 加入 Renovate ignore / packageRule 临时冻结
6. 复盘生成 Harness Patch 或 Rule Patch
```

示例冻结规则：

```json
{
  "packageRules": [
    {
      "matchPackageNames": ["example.com/broken/module"],
      "enabled": false,
      "description": "Temporarily frozen due to regression"
    }
  ]
}
```

## 19. Iteration / Self-improving Mechanism

每次 dependency PR 后自动生成复盘输入：

```text
- 哪些 Gate 失败？
```
